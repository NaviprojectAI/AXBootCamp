## 슬라이드 1 — 클로드 코드, 업무 자동화의 시작
"안녕하세요, '클로드 코드, 업무 자동화의 시작' 강의를 시작하겠습니다.
입문부터 고급까지 클로드 코드를 실무에 활용하는 방법을 함께 배워보겠습니다.
Mac 사용자분도 계시면 설치 부분만 조금 다르고 나머지는 동일하니 걱정 안 하셔도 됩니다."

## 슬라이드 2 — 이 강의를 시작하게 된 이유
"채팅형 AI 많이 쓰고 계시잖아요. 답변 정말 잘하죠. 문서도 만들어주고 다운로드도 됩니다.
그런데 제가 실제로 쓰면서 느낀 점을 말씀드리겠습니다.

이 표를 보시면 — 같은 업무인데 과정이 완전히 다릅니다.

좌측이 claude.ai 채팅형 AI를 사용했을때, 우측은 ClaudeCode를 사용했을때 입니다.


첫 번째 — 엑셀 파일 30개를 합쳐서 요약 보고서를 만들어야 했습니다.
채팅형 AI에서도 파일 여러 개를 올릴 수 있고 요약도 잘해 주는데 하지만 결과물을 다운로드하고, 내 폴더에 정리하는 건 직접 해야됩니다.

Claude Code에서는 '이 폴더에 있는 엑셀 전부 합쳐서 요약해' 한 마디면 결과 파일까지 같은 폴더에 바로 저장됩니다. 파일 업로드도, 다운받아서 정리할 필요도 없어요.

두 번째 — 프로젝트 산출물 관리입니다.
채팅형 AI에서 여러가지 프로젝트들을 진행하면서 기획서 나오면 다운로드하고 보고서 만들면 또 다운로드, 근데 수정 사항 생기면 각각 다시 다운로드 하면서 전부 제가 직접 했어야 했는데
Claude Code는 내 작업 폴더 안에서 직접 파일을 생성뿐만 아니라 기존 산출물이 있으면 알아서 파악하고 수정이 필요하면 바로 업데이트합니다.

세 번째 — 매주 같은 양식으로 보고서를 만들어야 했습니다.
채팅형 AI에서는 매번 '이런 양식으로, 이런 데이터를 넣어서 정리해줘'라고 처음부터 다시 설명해야 합니다.
Claude Code에서는 프로젝트 설정 파일에 양식과 규칙을 저장해두면, '보고서 써' 한 마디에 데이터만 바꿔서 바로 실행됩니다. 맥락을 다시 설명할 필요가 없습니다. (스킬)

네 번째 — 간단한 대시보드를 만들어달라고 했습니다.
채팅형 AI도 코드를 잘 만들어줍니다. 동작은 클로드 들어가서 아티팩트 환경에서만 동작이 되거나  간단한 HTML이 아닌 이상 잘 동작하지 않습니다. 

Claude Code에서는 코드 만들고 나서 '실행해' 한 마디면 알아서 설치하고, 서버 띄우고, 브라우저에서 바로 확인됩니다.

다섯 번째 — 만든 서비스를 외부에 배포하는 것도 마찬가지입니다.
채팅형 AI에서는 서버 세팅, 빌드, 배포를 전부 직접 해야 하지만,
Claude Code에서는 '배포해' 한 마디면 빌드부터 배포까지 한 번에 처리됩니다.

정리하면 — Claude Code를 활용하면 업무 자동화의 범위가 확장됩니다. 오늘 그 방법을 알려드리겠습니다."


## 슬라이드 3 — 클로드 코드 전환 기대 효과
"앞에서 본 차이를 좀 더 정리하면 이렇습니다.

표를 보시면 네 가지 효과가 있습니다.

첫째, 내 폴더에서 연속 작업 — 파일 업로드하고 맥락 다시 설명할 필요 없이, 프로젝트 폴더 안에서 바로 이어서 작업합니다.
둘째, 업무별 독립 공간 — 하나의 계정이라도 프로젝트마다 맥락이 분리됩니다. 지식이 섞이지 않아요.
셋째, 5배 큰 컨텍스트 — 대화가 끊기지 않아서 복잡한 작업도 한 세션에서 완료할 수 있습니다.
넷째, 만들고 바로 실행 — Artifact 미리보기가 아니라, 실제 환경에서 실행하고 빌드하고 배포까지 됩니다.

한 줄로 정리하면, 채팅형 AI가 '만들어주는 도구'라면, Claude Code는 '함께 일하는 환경'입니다."

## 슬라이드 4 — 클로드 코드 도입 사례 — 나라장터 입찰공모 자동화
"실제로 제가 클로드 코드로 자동화한 사례를 보여드리겠습니다.

AS-IS, 기존 방식입니다.
나라장터에서 키워드별로 직접 검색하고, 결과를 수동으로 정리했습니다.
관심 공고마다 상세 페이지를 직접 클릭해서 자료를 다운받고, 별도로 수집해야 했습니다. 상세 페이지 링크를 취득할 수가 없으니까, 나중에 다시 보려면 처음부터 검색해야 했어요.

TO-BE, 클로드 코드로 자동화한 방식입니다.
클로드 코드한테 자동화 프로그램을 만들어 달라고 의뢰했고, 완성된 프로그램으로 업무를 처리합니다. '오늘 입찰공고 검색해서 시트에 정리해줘' 한 마디면 끝입니다. 상세 링크도 자동으로 취득해서 저장하니까, 공고 상세 페이지에 바로 접근할 수 있습니다.

클로드 코드 내에서 자동화 프로그램을 만들어도 되고, 직접 명령을 한 문장 정도 치면 바로 자동화할 수도 있습니다."

"나라장터 데모 실행해"
"입찰 공고에서 컨설팅 키워드로 검색"
"XXXX 링크 알려줘"

## 슬라이드 5 — 팀에서 같은 계정을 쓴다면?
"팀에서 Claude 계정을 같이 쓰시는 분들도 계실 텐데요.

Claude.ai나 Desktop에서는 설정이 클라우드에 저장됩니다. 한 명이 설정을 바꾸면 전원에게 영향이 가요. 대화 기록도 섞여서 누가 뭘 했는지 구분이 어렵습니다.

Claude Code는 각자 PC에서 독립적으로 작동합니다. 설정 파일인 CLAUDE.md, MCP, Skill 같은 것들이 전부 로컬에 저장되니까 서로 영향이 없어요. 대화와 작업 이력도 각자의 터미널에 독립되어 있고, 프로젝트 폴더별로 완전히 분리됩니다.

같은 계정, 같은 요금제지만 Claude Code는 각자의 작업 환경이 보장됩니다."

## 슬라이드 6 — Claude 제품군 한눈에 비교
"Claude 제품군을 한눈에 비교해보겠습니다.

Claude.ai는 웹 브라우저에서 누구나 쓸 수 있는 형태입니다. 파일 수정이나 코드 실행은 불가능하고, 1회성 대화 위주입니다.

Claude Desktop, Cowork라고도 하는데요 — 데스크톱 앱으로 폴더 권한 설정 시 파일 수정이 가능하고, Extensions로 MCP도 간편하게 설치됩니다. 예약 작업이나 멀티스텝도 지원합니다. '쉽고 안전하게'가 컨셉입니다.

Claude Code는 CLI와 VSCode 확장으로 제공됩니다. 원래 개발자 대상이었지만 비개발자도 사용하는 추세예요. 파일 수정, 코드 실행이 자유롭고, Skill과 Hook과 MCP를 조합한 자동화가 가능합니다. '강력하고 자유롭게'가 컨셉입니다.

같은 클로드 두뇌지만, 몸, 즉 도구가 다릅니다."

## 슬라이드 9 — Step 1: Git 설치 (Windows 필수)
"첫 번째 단계는 Git 설치입니다. Windows 사용자분은 필수예요. Mac은 Git이 기본 내장되어 있어서 별도 설치가 필요 없습니다.

Git이 뭐냐면, 쉽게 말해서 자동 저장 + 되돌리기 도구입니다. Ctrl+Z를 폴더 전체 단위로 무한대로 해주는 것이라고 생각하시면 됩니다. '3일 전 상태로 돌아가줘' 하면 돌아갈 수 있어요.

Git이 왜 필요한지 두 가지 이유가 있습니다.
첫째, 클로드 코드가 사용하는 셸인 Git Bash가 Git과 함께 설치됩니다.
둘째, 클로드 코드가 파일을 수정하다 잘못되면 되돌릴 수 있는 안전망이 됩니다.

설치는 간단합니다. git-scm.com에 접속해서 'Download for Windows' 클릭하고, 대부분 기본값으로 Next 눌러서 진행하시면 됩니다. 설치 완료 후 터미널에서 `git --version` 입력해서 버전이 나오면 성공입니다."

## 슬라이드 10 — Step 2: Claude Code 설치
"두 번째, 클로드 코드 설치입니다.

두 가지 방법이 있는데, 추천하는 방법은 PowerShell에서 설치하는 겁니다. `irm https://claude.ai/install.ps1 | iex` 이 한 줄이면 됩니다. PowerShell 방식으로 설치하면 자동 업데이트도 지원됩니다.

WinGet으로도 설치할 수 있는데, Windows 11에는 기본 내장이고 Windows 10에서는 미설치일 수 있습니다.

혹시 PATH 경고가 나오면, 시스템 속성에서 환경 변수를 열고 사용자 PATH에 경로를 추가한 뒤 터미널을 재시작하시면 됩니다.

Mac이나 Linux 사용자분은 `curl -fsSL https://claude.ai/install.sh | bash`로 설치하시면 됩니다."

## 슬라이드 11 — Step 3: VSCode 설치 + Claude Code 확장
"세 번째, VSCode 설치와 Claude Code 확장 설치입니다.

VSCode는 code.visualstudio.com에서 다운로드받으시면 됩니다.
설치 후에 좌측 확장 아이콘을 클릭하거나 Ctrl+Shift+X를 누르고, 'Claude Code'를 검색해서 Anthropic 공식 확장을 설치합니다. 설치가 되면 좌측 사이드바에 Claude 아이콘이 나타납니다.

왜 VSCode 확장을 쓰냐면 — 파일 수정 내역을 에디터에서 바로 확인할 수 있고, diff 미리보기로 수정한 부분을 색상으로 한눈에 비교할 수 있어요. 파일 탐색기, 터미널, 에디터가 한 화면에 있어서 편합니다.

CLI가 모든 기능을 지원하지만, 이 강의에서는 VSCode 확장 기준으로 진행합니다."

## 슬라이드 12 — Step 4: 인증 설정
"네 번째, 인증 설정입니다.

추천하는 방법은 Claude Pro 또는 Max 구독입니다. claude.ai에서 구독하시고, 터미널에서 `claude`를 입력하면 브라우저 인증 창이 열립니다. 로그인하면 자동으로 연결됩니다.

두 번째 방법은 API 키입니다. console.anthropic.com에서 API 키를 발급받고, 사용한 만큼 비용이 청구되는 종량제 방식입니다."

## 슬라이드 13 — Step 5: 사용량 확인 — /usage
"설치 후 가장 먼저 해볼 것은 `/usage` 명령어입니다.

현재 사용량과 남은 한도를 확인할 수 있습니다. 토큰이라는 개념이 있는데, 클로드가 읽고 쓰는 모든 텍스트가 토큰으로 측정됩니다. 쉽게 말해 토큰은 돈입니다.

Pro나 Max 구독이면 월 정해진 한도 내에서 사용하고, API는 사용한 토큰만큼 과금됩니다.

습관적으로 작업 시작 전, 끝난 후에 `/usage` 확인하시는 걸 추천합니다."

## 슬라이드 18 — CLAUDE.md란?
"CLAUDE.md에 대해 알아보겠습니다.

클로드 코드를 실행하면 가장 먼저 CLAUDE.md 파일을 찾아서 통째로 읽습니다. 매 세션마다요.
이 파일에 프로젝트에서 지켜야 할 규칙과 맥락을 적어두면, 한 번 써두면 Claude가 매번 자동으로 기억합니다.

생성 방법은 `/init` 명령어입니다. 입력하면 현재 폴더를 분석해서 CLAUDE.md를 자동 생성해줍니다.
다만 빈 폴더에서는 /init 해도 거의 빈 파일이 나옵니다. 분석할 내용이 없으니까요. 프로젝트 파일이 좀 쌓인 상태에서 실행하는 게 좋습니다."

## 슬라이드 19 — CLAUDE.md 로딩 순서
"CLAUDE.md가 어떤 위치에서 읽히는지 보겠습니다.

로딩 순서가 세 단계입니다.
첫째, 시스템 프롬프트 — Anthropic이 제공하는 기본 설정서입니다.
둘째, CLAUDE.md — 사용자가 작성한 사전 지시서입니다.
셋째, 사용자 대화 — 우리가 입력하는 프롬프트입니다.

CLAUDE.md는 대화보다 먼저 읽히는 사전 지시서 역할을 합니다. 작성된 지시는 대화마다 자동 적용되니까, 반복 설명이 필요 없어요."

## 슬라이드 20 — 한 번 써두면 매번 자동 적용
"예시를 보겠습니다.

예시 1 — '코드에 주석을 꼭 달아줘'라고 CLAUDE.md에 적어두면, 매번 말 안 해도 알아서 주석을 추가합니다.
예시 2 — '새 파일 만들기 전에 나한테 먼저 확인해'라고 적어두면, 매번 먼저 물어보고 생성합니다.

역할도 넣을 수 있어요. '너는 사회적가치 컨설팅 전문가야. 보고서는 항상 한국어로, 존댓말로 작성해줘' 이렇게 역할부터 정해주면 일관된 톤으로 작업합니다.

역할, 규칙, 선호사항을 한 번 써두면 프로젝트 끝날 때까지 계속 적용됩니다."

## 슬라이드 21 — Context ≠ Enforcement
"한 가지 주의할 점이 있습니다.

CLAUDE.md는 시스템 프롬프트가 아니라 사용자 메시지로 전달됩니다. Claude가 읽고 따르려고 하지만, 모호한 지시는 무시될 수 있어요.

맥락이지 강제가 아닙니다. 구체적으로 쓰는 것이 핵심입니다."

## 슬라이드 22 — 나쁜 CLAUDE.md vs 좋은 CLAUDE.md
"나쁜 패턴과 좋은 패턴을 비교해보겠습니다.

첫 번째 나쁜 패턴 — 너무 긴 경우입니다. 300줄 이상이면 중요한 규칙이 노이즈에 묻혀서 Claude가 무시하기 시작합니다. 공식 권장은 200줄 이내, 실무 권장은 60줄 이내입니다.

두 번째 나쁜 패턴 — 모호한 지시입니다. '깔끔한 코드를 작성하세요'는 나쁘고, '함수는 30줄 이하로 작성하세요'가 좋습니다. 검증 가능한 구체적인 행동을 지시해야 합니다.

세 번째 나쁜 패턴 — 인덱싱 없이 그냥 나열하는 경우입니다. 내용이 길어지면 별도 .md 파일로 분리하고, CLAUDE.md에서는 참조만 남기는 게 효과적입니다. CLAUDE.md는 인덱스, 즉 목차 역할이고 상세 내용은 각 파일에 두는 거죠."

## 슬라이드 23 — /init — CLAUDE.md 자동 생성
"처음부터 직접 작성할 필요 없습니다.

프로젝트 폴더에서 `/init`을 실행하면 Claude가 프로젝트를 분석하고 CLAUDE.md를 자동으로 생성합니다.
프로젝트 구조, 사용 언어, 빌드 방식 등을 자동으로 파악해주고, 기본 골격이 생성되면 필요한 규칙을 추가하거나 수정하시면 됩니다."

## 슬라이드 24 — 컨텍스트란?
"컨텍스트에 대해 알아보겠습니다.

컨텍스트란 클로드가 지금 기억하고 있는 것의 범위입니다.
대화 내용, 읽은 파일, 실행한 명령어 — 전부 컨텍스트에 쌓입니다. 컨텍스트가 클수록 클로드가 더 정확하게 판단하지만, 무한하지는 않습니다.

컨텍스트 윈도우, 즉 한번에 볼 수 있는 크기가 현재 최대 1M 토큰, 책 약 5권 분량입니다.

비유를 하나 드리면 — 일일 전문가 직원이라고 생각하시면 됩니다. 컨텍스트가 꽉 차면 직원이 퇴근하고 새 직원이 출근하는 거예요. 다만 인수인계는 해줍니다. 핵심 내용은 넘어가지만 세부적인 건 다시 설명해야 할 수 있어요.

실전 팁으로는, 한 가지 주제에 집중해서 대화하고, 주제가 바뀌면 `/clear`로 새로 시작하고, 긴 작업은 단계별로 나눠서 요청하는 게 좋습니다."

## 슬라이드 25 — /clear — 대화 초기화
"주제가 바뀌거나 새로 시작하고 싶을 때 `/clear`를 사용합니다.

현재 대화 내용을 전부 지우고 새로운 대화를 시작합니다. 단, CLAUDE.md와 Auto Memory는 유지됩니다. 프로젝트 설정은 사라지지 않아요.
이전 대화의 실수나 혼선이 계속 영향을 줄 때 유용합니다.

비유하면 새 동료로 교체하는 건데, 업무 매뉴얼인 CLAUDE.md는 그대로 남아있는 겁니다."

## 슬라이드 26 — /compact — 대화를 압축하기
"대화가 길어졌지만 맥락을 유지하고 싶을 때는 `/compact`를 사용합니다.

/clear와 다릅니다. 대화를 지우지 않고 요약해서 압축합니다. 컨텍스트 윈도우 공간을 확보하면서 핵심 맥락은 유지하는 거죠.

/compact는 같은 직원한테 '핵심만 정리해' 하는 겁니다.

참고로 auto-compact가 있어서 컨텍스트가 꽉 차면 자동으로 돌아가긴 하지만, 꽉 찬 상태에서 하면 요약 품질이 떨어져요. 여유 있을 때 미리 /compact 해두는 게 좋습니다.

정리하면, /clear는 전부 지우고 새로 시작 — 새 동료로 교체하는 것이고, /compact는 핵심만 남기고 압축 — 같은 동료에게 핵심만 메모로 전달하는 것입니다.
작업 중이라면 /clear 대신 /compact가 더 안전합니다."

## 슬라이드 27 — Auto Memory — 대화를 넘어서 기억하기
"Auto Memory에 대해 알아보겠습니다.

대화가 끝나도 중요한 정보를 기억하는 기능입니다. 클로드가 대화 중 중요한 정보를 자동으로 저장하고, 다음 대화에서도 이전에 배운 내용을 활용합니다.

기억하는 것들로는 — 사용자 선호, 프로젝트 맥락, 피드백 같은 것들입니다. '이렇게 하지 마'라고 피드백을 주면 다음부터 반영합니다.

직접 요청도 가능합니다. '앞으로 모든 응답은 한국어로 해줘 — 이걸 기억해줘'라고 하면 저장됩니다.

CLAUDE.md는 프로젝트의 매뉴얼이고, Auto Memory는 클로드의 업무 일지라고 생각하시면 됩니다."

## 슬라이드 28 — 컨텍스트 관리 전략 정리
"지금까지 배운 컨텍스트 관리 도구를 정리하겠습니다.

컨텍스트는 현재 대화의 작업 기억으로 이번 대화만 유지됩니다.
/clear는 대화를 초기화하고, /compact는 대화를 압축하면서 맥락을 유지합니다.
CLAUDE.md는 프로젝트 규칙과 매뉴얼로 항상 유지되고, Auto Memory는 학습된 선호와 피드백으로 자동 저장됩니다.

실전 흐름은 이렇습니다.
프로젝트 시작하면 CLAUDE.md로 기본 규칙을 설정하고, 작업 중에 컨텍스트가 쌓입니다. 컨텍스트가 부족하면 /compact로 압축하고, 주제가 전환되면 /clear로 초기화합니다. 중요한 패턴은 Auto Memory가 자동으로 저장합니다.

이 5가지를 이해하면 클로드와 장기적으로 효율적인 협업이 가능합니다."

## 슬라이드 29 — 프롬프트 엔지니어링 vs 컨텍스트 엔지니어링
"프롬프트 엔지니어링과 컨텍스트 엔지니어링을 비교해보겠습니다.

프롬프트 엔지니어링은 AI에게 좋은 질문을 하는 기술입니다. '어떻게 물어볼까?'에 초점이 맞춰져 있어요.
컨텍스트 엔지니어링은 AI가 잘 일할 환경을 만드는 기술입니다. '무엇을 미리 알려줄까?'에 초점이 맞춰져 있습니다.

프롬프트 엔지니어링은 말을 잘 거는 기술이고, 컨텍스트 엔지니어링은 일잘하는 환경을 세팅하는 기술입니다.

지금까지 프롬프트 엔지니어링은 이미 잘 해오셨죠.
이 강의에서는 여기서 한 단계 더 나아가서, 컨텍스트 엔지니어링을 배우게 됩니다.
수영에 비유하면 — 프롬프트 엔지니어링은 자유형, 배영, 평영 같은 개별 영법을 익히는 것이고, 컨텍스트 엔지니어링은 그 영법들을 언제 쓸지 코스를 설계하는 겁니다.
고급 과정까지 마치고 나면, 단순히 질문을 잘 하는 게 아니라 AI가 알아서 잘 일하는 환경 자체를 설계할 수 있게 됩니다."

## 슬라이드 30 — VSCode에서 클로드 코드 다루기
"이번에는 VSCode에서 클로드 코드를 다루는 메뉴와 설정을 살펴보겠습니다."

## 슬라이드 31 — Shift+Tab — 클로드의 자유도 조절
"입력창에서 Shift+Tab을 누르면 3가지 모드가 순환됩니다.

기본 모드는 파일 수정이나 명령 실행마다 '할까요?' 하고 물어봅니다.
자동 편집 모드는 파일 편집은 자동 승인하고, 명령 실행은 물어봅니다.
플랜 모드는 읽기만 가능하고 편집이나 실행을 안 합니다. 계획만 세우는 모드입니다.

AI는 사용자 얘기를 듣고 바로 실행부터 하려고 해요. 저는 좀 더 대화하면서 방향을 맞추고 싶은데, 클로드가 먼저 파일을 수정해버리는 경우가 있거든요. 그래서 저는 플랜 모드를 자주 씁니다. '일단 뭘 할 건지 계획부터 말해봐' 하고 확인한 다음에 진행시키는 거죠.
신입한테 '바로 하지 말고 먼저 보고해' 하는 것과 같습니다."

## 슬라이드 32 — @ 파일 참조 & 이미지 붙여넣기
"@ 파일 참조와 이미지 붙여넣기 기능입니다.

@ 뒤에 파일명을 입력하면 특정 파일이나 폴더를 빠르게 지정할 수 있습니다. 파일명 일부만 입력해도 자동완성됩니다.
예를 들어 '@README.md 이 파일 내용 요약해줘'라고 하면 됩니다.

이미지 붙여넣기도 됩니다. Ctrl+V로 클립보드 이미지를 바로 붙여넣을 수 있어요. 에러 화면을 캡처해서 '이 에러 뭐야?' 하거나, UI 디자인 이미지를 보여주고 '이거랑 비슷하게 만들어줘' 할 수 있습니다.

텍스트로 설명하기 어려우면, 그냥 보여주세요."

## 슬라이드 33 — VSCode 메뉴 살펴보기
"클로드 코드 대화창에서 설정 메뉴를 열면 여러 항목이 있습니다.

Switch model은 사용할 모델을 변경하는 거고, Effort는 추론 깊이를 조절하는 겁니다. Thinking은 사고 과정 표시를 켜고 끌 수 있고, Account & usage로 사용량을 확인할 수 있습니다.

작업에 맞게 조절하면 토큰도 아끼고 속도도 올릴 수 있습니다."

## 슬라이드 34 — Model 전환 — 상황에 맞는 두뇌 선택
"모델 전환에 대해 알아보겠습니다.

Opus는 가장 똑똑하고 가장 비쌉니다. 복잡한 분석이나 중요한 작업에 씁니다.
Sonnet은 균형 잡혀 있어서 일반적인 작업 대부분에 적합합니다.
Haiku는 가장 빠르고 가장 저렴합니다. 간단한 질문이나 빠른 확인에 씁니다.

실전 전략은 — 기본은 Sonnet으로 작업하고, 복잡한 판단이 필요하면 Opus로 전환하고, 단순 확인은 Haiku로 토큰을 절약하는 겁니다.
모든 작업에 가장 비싼 모델을 쓰는 건 낭비입니다."

## 슬라이드 35 — Effort — 얼마나 깊이 생각할까?
"Effort 설정입니다.

Low는 간단한 작업에 빠르게 답변하고 토큰 소모가 적습니다.
Medium은 표준 추론으로 기본값입니다.
High는 깊이 생각하고 답변하며 토큰 소모가 높습니다.
Max는 최대 추론, 제약 없이 사고하며 토큰 소모가 매우 많습니다.

Effort는 '이 작업에 얼마나 시간을 들여라'라는 지시라고 생각하시면 됩니다."

## 슬라이드 36 — Thinking — 사고 과정 보기
"Thinking을 ON으로 켜면 클로드가 답변 전에 생각하는 과정을 보여줍니다.

클로드가 왜 그런 결정을 했는지 알고 싶거나, 복잡한 작업에서 방향이 맞는지 확인할 때 유용합니다. 학습 목적으로도 좋아요.

단순 반복 작업이나 빠른 응답이 필요할 때는 끄는 게 좋습니다.

Thinking ON은 토큰을 더 쓰지만, 더 정확한 결과를 얻을 수 있습니다."

## 슬라이드 37 — 설정 조합 가이드
"상황별 설정 조합을 정리하면 이렇습니다.

간단한 질문은 Sonnet + Low + Thinking OFF.
파일 정리는 Sonnet + Medium + Thinking OFF.
보고서 분석은 Sonnet + High + Thinking ON.
복잡한 설계는 Opus + Max + Thinking ON.

상황에 맞게 조합하면 비용은 줄이고 품질은 유지할 수 있습니다."

## 슬라이드 38 — 되돌리기
"클로드 코드는 실제로 파일을 수정합니다. 하지만 되돌리기가 가능합니다.

기본 안전장치로 Rewind가 있어서 클로드가 수정한 내용을 되돌릴 수 있습니다. 하지만 파일 삭제 같은 경우는 Rewind로도 복구가 어려울 수 있어요.

이럴 때 Git이 필요합니다. Git으로 커밋해두면 어떤 상황이든 복구 가능합니다."

## 슬라이드 39 — Git이란?
"Git에 대해 좀 더 알아보겠습니다.

한 마디로 파일의 저장 이력을 관리하는 도구입니다. 작업 폴더의 스냅샷을 찍어두는 도구라고 생각하시면 됩니다.

폴더 상태를 통째로 찍어두는 것을 커밋이라고 하고, 커밋이 계속 쌓이면 언제든 원하는 시점으로 이동할 수 있습니다. '3번째 사진으로 돌아가줘' 하듯이 '되돌려줘' 한 마디면 복구됩니다.

예전에는 명령어를 공부하고 별도 툴을 설치해야 했지만, 지금은 '커밋해', '되돌려줘' 자연어 한 줄이면 끝입니다.

마음껏 실험할 수 있는 안전망입니다."

## 슬라이드 40 — Chapter 3. 중급
"Chapter 3, 중급입니다. 확장과 자동화를 다루겠습니다.
이 챕터가 끝나면 Skill, MCP, Hook으로 클로드 코드를 업무에 맞게 확장할 수 있습니다.

이것들 없이도 자동화는 얼마든지 됩니다. 하지만 이 세 가지를 익히면 업무 자동화가 한층 더 가속돼요.

어려워 보여도 걱정하실 필요 없습니다. '이런 개념이 있구나' 정도만 이해하시면 되고, 전부 자연어로 '만들어줘'라고 하면 됩니다. 참고로 저도 클로드 코드를 매일 쓰지만, 직접 파일 열어서 코드를 수정하는 일은 거의 없어요. 다 말로 시킵니다."

## 슬라이드 41 — 이 챕터에서 배울 3가지
"이 챕터에서 배울 3가지를 먼저 보겠습니다.

Skill은 자주 하는 작업을 명령어로 저장하는 겁니다. 비유하면 매크로예요.
MCP는 외부 도구를 연결하는 겁니다. USB 포트와 같습니다.
Hook은 특정 이벤트에 자동으로 실행되는 겁니다. 알람 설정과 비슷해요.

이 3가지를 조합하면 나만의 자동화 시스템이 만들어집니다."

## 슬라이드 42 — Skill
"먼저 Skill에 대해 알아보겠습니다. AI에게 주는 업무 매뉴얼이라고 생각하시면 됩니다."

## 슬라이드 43 — Skill이란?
"Skill이란, AI에게 주는 업무 매뉴얼입니다. 한 번 만들면 자동으로 적용됩니다.

신입 사원에게 업무 매뉴얼을 주면 일정한 품질의 결과물이 나오듯, Skill을 만들어두면 클로드가 매번 같은 품질로 작업합니다.

프롬프트와의 차이를 보면 — 프롬프트는 매번 직접 입력해야 하고, 재사용하려면 복사 붙여넣기가 필요합니다. Skill은 한 번 만들면 이름만 부르면 자동으로 트리거되고, 파일 하나로 팀에 공유할 수 있습니다.

Skill은 내가 자주 시키는 일을 미리 적어둔 지시서입니다."

## 슬라이드 44 — Skill의 구조 — 두 단계 로딩
"Skill의 구조를 보겠습니다.

skill.md 파일이 진입점이고, 참조 파일과 함께 디렉토리 단위로 관리됩니다.
.claude/skills/ 폴더 아래에 각 Skill별 폴더가 있고, 그 안에 skill.md 파일이 들어있습니다.

똑똑한 점은 두 단계로 로딩한다는 겁니다. 1단계에서는 Description, 즉 설명만 로딩합니다. 매우 가볍습니다. 2단계에서는 필요할 때만 전체 프롬프트를 로딩합니다.

CLAUDE.md는 항상 로딩되어 무조건 컨텍스트를 차지하지만, Skill은 필요할 때만 로딩되니까 컨텍스트 부담이 적습니다.

비유하면 CLAUDE.md는 항상 펼쳐놓은 매뉴얼이고, Skill은 필요할 때 꺼내 보는 매뉴얼입니다."

## 슬라이드 45 — Skill 만들기 — 실전 예시
"실전 예시를 보겠습니다.

skill.md는 설정 정보와 본문 프롬프트로 구성됩니다. 설정에는 name과 description이 들어가고, 본문에는 실제 지시 내용이 들어갑니다.

Description이 핵심입니다. AI가 '이 Skill을 쓸지 말지' 판단하는 필드이기 때문에, 사용자가 실제로 말할 법한 표현을 포함해야 합니다.

예를 들어 '컨설팅 키워드로 공고 검색해서, 결과를 구글시트에 정리하고, 담당자한테 메일 보내줘' 이 한 문장이면 검색 Skill이 돌고, 시트에 기록하고, 메일까지 발송됩니다. 이게 자동화예요.

Skill Creator를 사용하면 대화로 쉽게 생성할 수 있습니다."

## 슬라이드 46 — Skill 생성 방법과 저장 위치
"Skill을 만드는 방법입니다.

권장하는 방법은 Skill Creator를 사용하는 겁니다. 클로드에게 'skill-creator 설치해줘'라고 하면 설치되고, /skill-creator를 실행하면 됩니다. 실행하면 '어떤 Skill을 만들고 싶으세요?' 하고 물어보면서 대화로 함께 설계해나갑니다.

저 같은 경우에는 아까 시연했던 나라장터 프로그램을 먼저 만들고, 그 기능을 Skill로 하나 만들어달라고 했어요. 그러면 Skill Creator가 기존 코드를 분석해서 적절한 skill.md를 만들어줍니다.

저장 위치에 따라 사용 범위가 달라집니다. 프로젝트 레벨인 `.claude/skills/`에 저장하면 해당 프로젝트에서만 사용되고, 유저 레벨인 `~/.claude/skills/`에 저장하면 모든 디렉토리에서 사용할 수 있습니다.
팀 공유는 프로젝트 레벨, 개인 생산성은 유저 레벨이 적합합니다."

## 슬라이드 47 — 실습: Skill 만들어보기
"Skill 실습을 해보겠습니다.

데모 프로젝트에서 만든 Skill을 먼저 확인합니다.
/bid-search는 키워드로 입찰공고를 검색하고, /bid-link는 공고 상세 페이지 링크를 조회하고, /bid-report는 오늘의 검색 결과 보고서를 생성하고, /bid-sync는 Google Sheets에 결과를 동기화합니다.

demo/.claude/skills/ 폴더를 열어서 bid-search/skill.md 파일의 구조를 살펴보세요. name, description 필드의 역할과, 본문 프롬프트가 어떻게 작성되었는지 확인합니다.

그다음 '입찰 공고에서 컨설팅 키워드로 검색해줘'라고 말해서 /bid-search Skill이 트리거되는지 확인해보세요.

직접 만들어보려면 /skill-creator를 실행해서 '입찰공고 일일 보고서를 만드는 Skill을 만들어줘'라고 요청하시면 됩니다."

## 슬라이드 48 — MCP
"다음은 MCP입니다. 외부 도구를 연결하는 기능입니다."

## 슬라이드 49 — MCP란?
"MCP, Model Context Protocol입니다. 클로드와 외부 도구를 연결하는 표준이에요.

클로드 코드 혼자서는 노션 페이지 수정, 구글 시트 수정, 슬랙 메시지 전송이 안 됩니다.
MCP를 연결하면 이 세 가지가 전부 가능해집니다.

비유하면 스마트폰에 앱을 설치하는 것과 같습니다. MCP는 클로드의 앱 스토어인 셈이죠.

다만 최근에는 MCP가 하향세인 부분도 있습니다. 이유가 몇 가지 있는데요.
첫째, 컨텍스트를 너무 많이 먹습니다. 설치된 MCP의 모든 도구 설명이 시작할 때 로딩되거든요.
둘째, 서버 연결이 자주 끊기고 수동으로 재시작해야 하는 경우가 있습니다.
셋째, MCP를 굳이 안 써도 Skill을 통해서 API를 직접 호출하면 되는 경우가 많습니다. 그래도 노션이나 구글시트처럼 자주 쓰는 서비스는 MCP가 편리합니다."

## 슬라이드 50 — MCP 연결하기
"MCP 연결 방법입니다.

클로드에게 그냥 '노션 MCP를 연결해줘'라고 말하면 됩니다. 클로드가 자동으로 필요한 MCP 서버를 파악하고, 설정 파일을 생성하고, 연결 테스트까지 합니다.

수동 설정도 가능합니다. .mcp.json 파일에 직접 서버 정보를 입력하는 방식인데, 자연어 요청이 안 될 때 참고하시면 됩니다."

## 슬라이드 51 — 많이 쓰는 MCP 서버
"많이 쓰는 MCP 서버를 보면 — Notion, Google Drive, Slack, GitHub, Gmail 등이 있습니다.

실전 워크플로우 예시로, '이 엑셀 데이터를 분석해서 노션 보고서 페이지에 정리해줘'라고 하면 — 엑셀 읽기는 로컬에서, 분석은 클로드가, 노션 작성은 MCP가 처리합니다."

## 슬라이드 52 — ⚠️ MCP 설치 전략 — 적게, 핵심만
"MCP 설치 전략을 알려드리겠습니다.

MCP를 많이 설치하면 안 됩니다. 설치된 MCP의 모든 도구 설명이 시작 시 로딩되어 컨텍스트를 압박하고, 도구가 많을수록 클로드가 잘못된 도구를 선택할 확률이 올라갑니다.

최근 트렌드는 MCP 대신 Skill + Sub-Agent 조합으로 대체하는 경우가 늘고 있습니다.

권장 전략은 — 핵심 1~2개만 설치하고, 프로젝트별로 분리하고, 대체 가능하면 Skill로 하고, 안 쓰는 MCP는 비활성화하는 겁니다.

앱을 100개 설치한 스마트폰보다 필요한 앱 3개만 설치한 스마트폰이 빠릅니다."

## 슬라이드 53 — 실습: MCP 체험하기
"MCP 실습을 해보겠습니다.

실습 1에서는 구글시트 데이터를 읽어봅니다. '26공모 시트에서 3월 16~22일 리스트 알려줘'라고 하면 MCP가 시트를 읽고 표로 정리해줍니다.

실습 2에서는 구글시트를 수정합니다. 입찰공고 검색 결과를 '26공모 시트에 반영해줘'라고 하면 MCP가 마지막 행 다음에 데이터를 추가합니다.

실습 3에서는 구글시트 탭을 정리합니다. '26근무, 26공모 두 탭만 남기고 나머지 전부 제거해줘'라고 하면 클로드가 탭 목록을 조회하고 불필요한 탭을 자동 삭제합니다.

핵심 포인트는, 코드 한 줄 없이 자연어만으로 구글시트를 읽고 수정하고 삭제할 수 있다는 겁니다. 단, MCP는 항상 컨텍스트에 상주하니까 도구가 많을수록 무거워집니다."

## 슬라이드 54 — Hook
"마지막으로 Hook입니다. 클로드 코드의 자동화 엔진이에요."

## 슬라이드 55 — Hook이란?
"Hook이란, 이벤트가 발생하면 자동으로 액션을 실행하는 자동화 엔진입니다.

이벤트 발생 → 훅 감지 → 조건 검사 → 액션 실행, 이런 흐름입니다.

예를 들어, 파일 저장할 때 자동 포맷팅, 도구 호출 전 API 키 포함 여부 체크, 클로드 응답 대기 시 알림 전송, 작업 종료 시 결과 보고서 생성 같은 것들이 가능합니다.

비유하면 스마트홈의 '현관문 열리면 조명 켜기'나, 이메일의 '특정 발신자 → 자동 폴더 분류'와 같습니다.

Hook은 '조건 → 동작'의 자동화 규칙입니다."

## 슬라이드 56 — Hook의 4가지 이벤트 타입
"Hook에는 4가지 이벤트 타입이 있습니다.

PreToolUse는 도구 호출 직전에 실행됩니다. 입력값 검증이나 승인, 차단, 수정에 쓰입니다.
PostToolUse는 도구 실행 직후에 실행됩니다. 결과 검증이나 포맷팅에 쓰입니다.
Notification은 사용자 응답 대기 시 실행됩니다. 알림 전송이나 로깅에 쓰입니다.
Stop은 에이전트 턴이 종료될 때 실행됩니다. 최종 정리나 보고서 생성에 쓰입니다.

작동 흐름 예시를 보면 — 클로드가 파일 쓰기를 시도하면, PreToolUse에서 'API 키 포함?' 체크를 하고, 통과하면 파일 쓰기를 실행하고, PostToolUse에서 자동 포맷팅을 실행합니다."

## 슬라이드 57 — Hook 설정하기
"Hook을 설정하는 방법은 세 가지입니다.

첫째, `/hooks` 명령어로 현재 설정된 훅 목록을 확인하고 관리할 수 있습니다.
둘째, 클로드에게 '파일 저장할 때마다 자동으로 포맷팅하는 Hook 만들어줘'라고 요청하면 됩니다.
셋째, settings.json 파일을 직접 수정할 수도 있습니다.

생성된 훅은 settings.json에 저장되며, 이벤트 타입과 매칭 조건과 실행 액션으로 구성됩니다."

## 슬라이드 58 — Hook 활용 예시
"Hook 활용 예시 네 가지를 보겠습니다.

첫째, API 키 유출 차단입니다. PreToolUse로, 클로드가 파일을 쓰기 직전에 API 키나 비밀번호 포함 여부를 검사합니다. 나라장터 API 키가 코드에 하드코딩되면 쓰기 자체를 차단합니다.

둘째, 구글시트 기록 후 검증입니다. PostToolUse로, MCP로 구글시트에 공고를 추가한 직후에 행 번호와 데이터 정합성을 로그로 기록합니다.

셋째, 검색 완료 알림입니다. Notification으로, /bid-search로 크롤링 중 클로드가 대기할 때 알림음을 보냅니다. 다른 일을 하다가도 '검색 끝났어요' 알림으로 바로 복귀할 수 있어요.

넷째, 시트 기록 후 메일 제안입니다. PostToolUse로, Google Sheets MCP로 기록이 완료되면 '메일로 보내드릴까요?' 하고 자동으로 제안합니다.

Skill이 '무엇을' 하고, MCP가 '어디에' 연결하고, Hook이 '언제' 자동 실행할지 결정합니다."

## 슬라이드 59 — 실습: Hook 만들어보기
"Hook 실습입니다. 입찰공고 검색에서 시트 기록, 메일 제안까지 연결해보겠습니다.

프롬프트 한 줄이면 됩니다. '입찰공고에서 컨설팅 검색해서 26공모 시트에 기록해줘'

시나리오를 보면, /bid-search가 실행되어 컨설팅 키워드로 검색합니다. 이것이 Skill입니다.
그다음 '시트에 정리해줘' 하면 Google Sheets MCP로 기록합니다. 이것이 MCP입니다.
시트 기록이 완료되면 PostToolUse Hook이 작동해서 '메일로 보내드릴까요?' 하고 자동으로 제안합니다. 이것이 Hook입니다.
'응'이라고 하면 검색 결과 요약 메일이 발송됩니다. 다시 MCP입니다.

Skill이 작업을 수행하고, MCP가 외부 서비스에 연결하고, Hook이 다음 행동을 제안합니다."

## 슬라이드 60 — Skill + MCP + Hook = 나라장터 자동화 파이프라인
"지금까지 배운 세 가지를 한 장으로 정리하겠습니다.

'입찰공고에서 컨설팅 키워드로 검색해서 시트에 정리해줘' 이 한 마디에 일어나는 내부 동작입니다.

1번, /bid-search가 실행되어 키워드별 공고를 검색합니다. Skill입니다.
2번, 검색 결과를 Google Sheets에 기록합니다. MCP입니다.
3번, 시트 기록이 완료되면 '메일로 보내드릴까요?' 하고 제안합니다. Hook입니다.
4번, 승인하면 검색 결과 요약 메일을 발송합니다. 다시 MCP입니다.

Skill이 수행하고, MCP가 연결하고, Hook이 다음 행동을 제안하고, MCP가 또 연결합니다."

# 04. 고급

## 슬라이드 61 — Chapter 4. 고급
"Chapter 4, 고급입니다. 더 깊이 들어가보겠습니다.
이 챕터가 끝나면 멀티 에이전트와 CI/CD를 이해하고, 바이브 코딩의 함정을 알 수 있습니다."

## 슬라이드 62 — Sub-Agent란?
"Sub-Agent에 대해 알아보겠습니다.

메인 클로드 안에서 별도의 작업 공간을 가진 도우미 클로드를 생성하는 겁니다.

일반 모드에서는 나 → 클로드 1명 → 작업 → 결과, 이렇게 모든 맥락이 하나에 쌓입니다.
Sub-Agent 모드에서는 나 → 메인 클로드가 매니저 역할을 하고, Sub-Agent A는 자료 조사, B는 초안 작성, C는 검토를 담당합니다. 메인 클로드에는 요약 결과만 반환됩니다.

메인 클로드의 컨텍스트를 오염시키지 않고 독립적으로 작업한다는 것이 핵심입니다.

비유하면 팀장이 팀원에게 일을 나눠주고 보고서만 받는 겁니다. 팀원의 작업 과정은 팀장 머릿속에 안 남아요."

## 슬라이드 63 — Sub-Agent의 핵심 장점
"Sub-Agent의 핵심 장점은 네 가지입니다.

병렬 처리 — 여러 작업을 동시에 실행해서 전체 시간을 단축합니다.
컨텍스트 보호 — 메인 대화의 맥락을 오염시키지 않습니다.
전문화 — 각 에이전트에 전문 역할을 부여할 수 있습니다.
재사용 — 한 번 만든 에이전트를 계속 활용하고 공유할 수 있습니다.

특히 컨텍스트 보호가 중요합니다. 긴 대화 중에 조사 작업을 시키면 메인 맥락이 흐려지는데, Sub-Agent는 이걸 방지합니다."

## 슬라이드 64 — 언제 Sub-Agent를 쓸까?
"언제 Sub-Agent를 쓰면 좋을까요?

적합한 경우는 — 독립적인 작업이 여러 개일 때, 대량 파일을 탐색하고 분석할 때, 조사와 수정을 동시에 해야 할 때입니다.

반면 메인 대화가 적합한 경우도 있습니다. 단일 스레드로 순차 처리해야 하거나, 모든 맥락을 공유해야 하거나, 간단한 작업일 때는 Sub-Agent를 쓰면 오히려 토큰 낭비입니다.

일손이 필요할 때만 팀원을 부르세요."

## 슬라이드 65 — Sub-Agent 활용 패턴 4가지
"Sub-Agent 활용 패턴 4가지를 보겠습니다.

격리 패턴은 독립된 공간에서 작업 후 결과를 반환합니다. '이 폴더 분석해서 요약만 알려줘'처럼요.
병렬 패턴은 여러 에이전트를 동시에 실행합니다. '3개 파일을 동시에 검토해줘'처럼요.
체인 패턴은 A의 결과를 B에 전달하는 순차 처리입니다. '조사 → 초안 → 검토' 파이프라인이죠.
재개 패턴은 중단된 작업을 이어서 진행합니다.

비개발자분들이 가장 많이 쓰는 건 격리 패턴입니다. 조사를 시키고 결과만 받는 방식이에요."

## 슬라이드 66 — 클로드 코드의 내장 Sub-Agent
"사실 클로드 코드는 이미 내부적으로 Sub-Agent를 활용하고 있습니다.

Explorer는 코드베이스를 탐색하고 파일을 검색하고, Plan은 작업 계획을 수립하고, Bash는 터미널 명령어를 실행하고, Guide는 사용법을 안내합니다.

클로드가 알아서 필요한 내장 에이전트를 호출하니까 사용자가 직접 신경 쓸 필요 없습니다.

이미 여러분은 Sub-Agent의 혜택을 받고 있었습니다. 클로드가 파일을 찾고 계획을 세울 때마다 내부적으로 사용 중이었어요."

## 슬라이드 67 — 커스텀 에이전트 만들기
"나만의 에이전트를 만들 수도 있습니다.

'제안서 검토 전문 에이전트를 만들어줘'라고 하면 클로드가 역할, 사용 도구, 모델, 메모리 활성화 여부를 설정합니다.

생성된 에이전트는 .claude/agents/ 폴더에 agent.md 파일로 저장됩니다.
저장 위치에 따라 범위가 달라집니다. 프로젝트 폴더에 두면 해당 프로젝트에서만, ~/.claude/agents/에 두면 어디서든 사용 가능합니다."

## 슬라이드 68 — Agent Team — 에이전트끼리 협업하기
"Agent Team은 여러 클로드 인스턴스가 팀으로 협업하는 기능입니다.

Sub-Agent와의 차이를 보면 — Sub-Agent는 메인에게만 보고하는 상하 관계인 반면, Agent Team은 팀원 간에 직접 메시지를 교환합니다. 실제 개발팀과 유사한 협업 구조예요.

비유하면, Sub-Agent는 외주 맡기기이고 Agent Team은 프로젝트 팀을 구성하는 겁니다."

## 슬라이드 69 — CI/CD와 Claude Code
"CI/CD와 Claude Code에 대해 알아보겠습니다. 자동화의 끝판왕이라고 할 수 있습니다."

## 슬라이드 70 — CI/CD란?
"CI/CD란, 코드를 바꿀 때마다 자동으로 검사하고 배포하는 시스템입니다.

CI는 Continuous Integration, 자동 품질 검사이고, CD는 Continuous Deployment, 자동 배포입니다.

흐름을 보면 — 코드 수정 → 자동 테스트 → 테스트 통과 → 자동 배포. 실패하면 알림과 수정 요청이 갑니다.

비유하면 공장의 품질 검사 라인입니다. 불량품은 자동으로 걸러지는 거죠."

## 슬라이드 71 — Claude Code + CI/CD
"클로드 코드를 CI/CD에 넣으면 어떻게 되냐면,

코드를 변경해서 GitHub에 Push하면, CI가 자동 실행되어 클로드에게 코드 리뷰를 요청합니다. 클로드가 리뷰 결과를 코멘트하고, 문제 없으면 자동 배포됩니다.

사람이 리뷰 안 해도 기본 품질이 보장되고, 24시간 자동으로 동작하며, 일관된 기준으로 검사합니다.

이 부분은 개발팀과 협업할 때 알아두면 좋은 개념입니다."

## 슬라이드 72 — ⚠️ API 키 보안 — 신용카드처럼 관리하세요
"API 키 보안에 대해 중요한 내용을 말씀드리겠습니다.

API 키란 외부 서비스에 접근하는 디지털 열쇠입니다. Anthropic API 키, 노션 API 키, 나라장터 API 키 같은 것들이죠.
이 키가 유출되면 다른 사람이 내 돈으로 서비스를 사용할 수 있습니다.

실제 피해 사례로 — API 키가 GitHub에 노출되어 수십만에서 수백만 원이 과금되거나, 클라우드 서비스 키가 도난되어 암호화폐 채굴에 악용된 경우가 있습니다.

반드시 지켜야 할 규칙 세 가지입니다.
첫째, API 키는 .env 파일에 저장합니다.
둘째, .env 파일은 .gitignore에 추가해서 절대 Git에 커밋하지 않습니다.
셋째, 키가 노출된 것 같으면 즉시 폐기하고 재발급합니다.

API 키는 내 신용카드입니다. 유출은 곧 금전 피해입니다. 클로드에게 '이 키 커밋하지 마'라고 CLAUDE.md에 적어두세요."

## 슬라이드 73 — 제품 출시는 개발 이해가 필요합니다
"클로드 코드로 비개발자도 많은 것을 만들 수 있습니다. 하지만 '작동한다'와 '출시해도 된다'는 다릅니다.

웹페이지나 대시보드를 만들 수 있지만, 보안 설정과 접근 제어가 추가로 필요합니다. 자동화 스크립트도 만들 수 있지만, 에러 처리와 예외 상황 대응이 필요해요.

보안 이슈는 터지면 기업에 큰 피해가 됩니다. 개인정보 유출, 금전 손실 같은 것들이죠.

그래서 내부 업무 도구는 클로드 코드로 충분하고, 외부 공개 서비스는 개발자 검토가 필수이고, 보안이 중요한 영역은 반드시 전문가 확인이 필요합니다.

AI가 만들어준 결과물의 최종 책임은 사람에게 있습니다."

## 슬라이드 74 — ⚠️ 바이브 코딩 경고
"바이브 코딩에 대한 경고입니다. AI 과신의 위험에 대해 말씀드리겠습니다."

## 슬라이드 75 — 바이브 코딩이란?
"바이브 코딩이란, 분위기로 코딩하기입니다.

'대충 이런 느낌으로 만들어줘' → 클로드가 만들어요 → 작동하네? 끝! → 내용은 안 봄. 여기가 문제입니다.

겉보기엔 작동하지만 내부에 문제가 숨어있을 수 있습니다. 보안 취약점이나 데이터 손실 가능성이 있고, 나중에 문제가 터지면 원인 파악이 불가능합니다.

작동한다와 올바르다는 다릅니다."

## 슬라이드 76 — 올바른 AI 활용 vs 바이브 코딩
"올바른 AI 활용과 바이브 코딩을 비교하면 이렇습니다.

올바른 활용은 구체적으로, 단계별로 요청하고, 결과를 검토하고 이해하며, Git으로 버전 관리를 합니다.
바이브 코딩은 '대충 만들어줘'로 요청하고, 작동하면 OK, 뭐가 바뀌었는지도 모릅니다.

위험도 차이가 확실합니다."

## 슬라이드 77 — AI에게 일을 맡기는 올바른 자세
"AI에게 일을 맡기는 올바른 자세 네 가지입니다.

첫째, 구체적으로 요청하기. '홈페이지 만들어줘'가 아니라 '회사 소개 페이지를 만들어줘. 메뉴는 소개, 서비스, 연락처 3개'처럼요.

둘째, 결과를 확인하기. 파일이 만들어지면 열어서 확인하고, 모르는 부분은 '이거 왜 이렇게 했어?' 라고 물어보세요.

셋째, 단계별로 진행하기. 한번에 완성하려 하지 말고, 작은 단위로 나눠서 확인하며 진행합니다.

넷째, 안전망 유지하기. Git 커밋을 자주 하고, 중요한 작업 전에 백업합니다.

AI는 도구입니다. 최종 판단은 항상 사람의 몫입니다."

## 슬라이드 78 — 실습 4: 안전한 AI 활용 연습
"안전한 AI 활용 실습을 해보겠습니다.

먼저 나쁜 예를 체험합니다. '아무 웹사이트 하나 만들어줘'라고 요청해보세요. 결과를 확인합니다. 뭐가 만들어졌는지 이해가 되시나요?

그다음 좋은 예를 체험합니다.
'index.html 파일을 만들어줘. 내용은 자기소개 페이지'
'제목은 홍길동의 소개, 섹션은 인사말, 경력, 연락처 3개'
결과가 요청과 일치하는지 확인합니다.
'인사말 섹션의 문구를 바꿔줘'로 수정해보고, /undo로 되돌려봅니다.

차이가 느껴지시죠?"
