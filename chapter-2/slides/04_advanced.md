# Sub-Agent란?

**메인 클로드 안에서 별도의 작업 공간을 가진 "도우미 클로드"를 생성하는 것**

일반 모드:
```
나 → 클로드 1명 → 작업 → 결과
     (모든 맥락이 하나에 쌓임)
```

Sub-Agent 모드:
```
나 → 메인 클로드(매니저)
       ├→ Sub-Agent A: 자료 조사
       ├→ Sub-Agent B: 초안 작성
       └→ Sub-Agent C: 검토
     ← 요약 결과만 반환
```

- 메인 클로드의 **컨텍스트를 오염시키지 않고** 독립적으로 작업
- 결과만 깔끔하게 돌려받음

> 비유: 팀장이 팀원에게 일을 나눠주고, **보고서만 받는 것** — 팀원의 작업 과정은 팀장 머릿속에 안 남음

[슬라이드 63] [내용]
# Sub-Agent의 핵심 장점

| 장점 | 설명 | 비유 |
|------|------|------|
| **병렬 처리** | 여러 작업을 동시에 실행 → 전체 시간 단축 | 팀원 3명이 동시에 작업 |
| **컨텍스트 보호** | 메인 대화의 맥락을 오염시키지 않음 | 팀원 작업이 팀장 머릿속에 안 쌓임 |
| **전문화** | 각 에이전트에 전문 역할 부여 가능 | 조사 담당, 작성 담당, 검토 담당 |
| **재사용** | 한 번 만든 에이전트를 계속 활용·공유 | 매뉴얼화된 업무 프로세스 |

> 특히 **컨텍스트 보호**가 중요합니다 — 긴 대화 중 조사 작업을 시키면 메인 맥락이 흐려지는데, Sub-Agent는 이걸 방지합니다

[슬라이드 64] [내용]
# 언제 Sub-Agent를 쓸까?

**Sub-Agent가 적합한 경우:**
| 상황 | 이유 |
|------|------|
| 독립적인 작업 여러 개 | A, B, C가 서로 의존하지 않을 때 |
| 대량 파일 탐색·분석 | 메인 컨텍스트에 결과만 받으면 될 때 |
| 조사 + 수정 동시 | 하나는 조사하고, 하나는 수정 |

**메인 대화가 적합한 경우:**
- 단일 스레드로 순차 처리해야 할 때
- 모든 맥락을 공유해야 할 때
- 간단한 작업 (Sub-Agent 쓰면 오히려 토큰 낭비)

**주의사항:**
- 의존성 있는 작업의 병렬화는 지양
- 하나의 에이전트에 너무 많은 역할 부여 ❌
- 결과 검증 없이 사용하는 것은 비추천

> "일손이 필요할 때만 팀원을 부르세요"

# Sub-Agent 활용 패턴 4가지

| 패턴 | 설명 | 예시 |
|------|------|------|
| **격리 패턴** | 독립된 공간에서 작업 후 결과 반환 | "이 폴더 분석해서 요약만 알려줘" |
| **병렬 패턴** | 여러 에이전트 동시 실행 | "3개 파일을 동시에 검토해줘" |
| **체인 패턴** | A의 결과를 B에 전달, 순차 처리 | "조사 → 초안 → 검토" 파이프라인 |
| **재개 패턴** | 중단된 작업을 이어서 진행 | "아까 하던 분석 이어서 해줘" |

**가장 많이 쓰는 패턴:**
- 비개발자: **격리 패턴** (조사 시키고 결과만 받기)
- 실무자: **병렬 패턴** (여러 문서 동시 처리)

# 클로드 코드의 내장 Sub-Agent

**클로드 코드는 이미 내부적으로 Sub-Agent를 활용하고 있습니다:**

| 내장 에이전트 | 역할 |
|--------------|------|
| **Explorer** | 코드베이스 탐색, 파일 검색 |
| **Plan** | 작업 계획 수립 |
| **Bash** | 터미널 명령어 실행 |
| **Guide** | 사용법 안내 |

- 클로드가 알아서 필요한 내장 에이전트를 호출
- 사용자가 직접 신경 쓸 필요 없음

> 이미 여러분은 Sub-Agent의 혜택을 받고 있었습니다 — 클로드가 파일을 찾고, 계획을 세울 때마다 내부적으로 사용 중

# Agent Team — 에이전트끼리 협업하기

**여러 클로드 인스턴스가 "팀"으로 협업하는 기능**

```
리더 에이전트
  ├→ 팀원 A: 자료 수집     ←→ 직접 소통 가능
  ├→ 팀원 B: 초안 작성     ←→ 직접 소통 가능
  └→ 팀원 C: 검토·수정     ←→ 직접 소통 가능
```

**Sub-Agent와의 차이:**

| | Sub-Agent | Agent Team |
|---|----------|------------|
| **소통** | 메인에게만 보고 | 팀원 간 직접 메시지 교환 |
| **구조** | 상하 관계 | 실제 개발팀과 유사한 협업 |
| **용도** | 독립적 작업 분배 | 상호 의존적 복잡한 프로젝트 |

> 비유: Sub-Agent = 외주 맡기기, Agent Team = 프로젝트 팀 구성

# CI/CD란?

**코드를 바꿀 때마다 자동으로 검사하고 배포하는 시스템**

| 용어 | 풀네임 | 비유 |
|------|--------|------|
| **CI** | Continuous Integration | 자동 품질 검사 |
| **CD** | Continuous Deployment | 자동 배포 |

**예시 흐름:**
```
코드 수정 → 자동 테스트 → 테스트 통과 → 자동 배포
               ↓ 실패
          알림 + 수정 요청
```

> 비유: 공장의 "품질 검사 라인". 불량품은 자동으로 걸러짐

# Claude Code + CI/CD

**클로드 코드를 CI/CD에 넣으면:**

1. 코드 변경 → GitHub에 Push
2. CI가 자동 실행 → 클로드에게 코드 리뷰 요청
3. 클로드가 리뷰 결과 코멘트
4. 문제 없으면 자동 배포

**이게 왜 강력한가?**
- 사람이 리뷰 안 해도 기본 품질 보장
- 24시간 자동 동작
- 일관된 기준으로 검사

> 이 부분은 개발팀과 협업할 때 알아두면 좋은 개념입니다

# ⚠️ API 키 보안 — 신용카드처럼 관리하세요

**API 키란?** 외부 서비스에 접근하는 "디지털 열쇠"
- Anthropic API 키, 노션 API 키, 나라장터 API 키…
- 이 키가 유출되면 → **다른 사람이 내 돈으로 서비스를 사용**

**실제 피해 사례:**
- API 키가 GitHub에 노출 → 수십만~수백만 원 과금
- 클라우드 서비스 키 도난 → 암호화폐 채굴에 악용
- 개인 신용카드를 인터넷에 올리는 것과 동일한 위험

**반드시 지켜야 할 규칙:**
1. API 키는 `.env` 파일에 저장
2. `.env` 파일은 `.gitignore`에 추가 → **절대 Git에 커밋하지 않기**
3. 키가 노출된 것 같으면 → **즉시 폐기하고 재발급**

```
# .env 파일 (이 파일은 Git에 올리면 안 됨!)
NOTION_API_KEY=ntn_xxxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxx
```

> API 키 = 내 신용카드. **유출 = 금전 피해**. 클로드에게 "이 키 커밋하지 마"라고 CLAUDE.md에 적어두세요

# 제품 출시는 개발 이해가 필요합니다

**클로드 코드로 비개발자도 많은 것을 만들 수 있습니다. 하지만…**

| 만들 수 있는 것 | 추가로 필요한 것 |
|----------------|----------------|
| 웹페이지, 대시보드 | 보안 설정, 접근 제어 |
| 자동화 스크립트 | 에러 처리, 예외 상황 대응 |
| 데이터 분석 도구 | 개인정보 보호, 데이터 보안 |
| API 연동 서비스 | 키 관리, 인증 체계 |

**"작동한다"와 "출시해도 된다"는 다릅니다:**
- 보안 이슈는 터지면 **기업에 큰 피해** — 개인정보 유출, 금전 손실
- 기능 설계는 **어떤 기능을 어떻게 조합할지** 아는 사람이 해야 효율적
- 운영/유지보수는 개발적 이해가 있어야 가능

**그래서:**
- 내부 업무 도구 → 클로드 코드로 충분
- 외부 공개 서비스 → 개발자 검토 필수
- 보안이 중요한 영역 → 반드시 전문가 확인

> AI가 만들어준 결과물의 **최종 책임은 사람**에게 있습니다

# 바이브 코딩이란?

**"분위기(vibe)로 코딩하기"**

```
"대충 이런 느낌으로 만들어줘"
→ 클로드가 만듦
→ 작동하네? 끝!
→ 내용은 안 봄 ← ⚠️ 여기가 문제
```

**위험한 이유:**
- 겉보기엔 작동하지만 내부에 문제가 숨어있을 수 있음
- 보안 취약점, 데이터 손실 가능성
- 나중에 문제 터지면 원인 파악 불가

> "작동한다 ≠ 올바르다"

# 올바른 AI 활용 vs 바이브 코딩

| | 올바른 활용 | 바이브 코딩 |
|---|-----------|------------|
| **요청** | 구체적, 단계별 | "대충 만들어줘" |
| **확인** | 결과를 검토하고 이해 | 작동하면 OK |
| **되돌리기** | Git으로 버전 관리 | "뭐가 바뀌었지?" |
| **학습** | 왜 이렇게 했는지 이해 | 관심 없음 |
| **위험** | 낮음 | ⚠️ 높음 |

# AI에게 일을 맡기는 올바른 자세

**1. 구체적으로 요청하기**
- ❌ "홈페이지 만들어줘"
- ✅ "회사 소개 페이지를 만들어줘. 메뉴는 소개, 서비스, 연락처 3개"

**2. 결과를 확인하기**
- 파일이 만들어지면 열어서 확인
- 모르는 부분은 "이거 왜 이렇게 했어?" 라고 물어보기

**3. 단계별로 진행하기**
- 한번에 완성하려 하지 말고
- 작은 단위로 나눠서 확인하며 진행

**4. 안전망 유지하기**
- Git 커밋 자주 하기
- 중요한 작업 전에 백업

> AI는 도구입니다. **최종 판단은 항상 사람의 몫**
