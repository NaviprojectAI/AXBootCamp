# CLAUDE.md란?

- 클로드 코드를 실행하면 가장 먼저 CLAUDE.md를 찾아서 **통째로 읽는다** (매 세션마다)
- 프로젝트에서 지켜야 할 **규칙과 맥락**을 적어두는 메모 파일
- 한 번 써두면 Claude가 매번 자동으로 기억합니다

**생성 방법: `/init` 명령어**
- `/init` 입력 → 현재 폴더를 분석해서 CLAUDE.md를 자동 생성

# CLAUDE.md 로딩 순서

| 순서 | 단계 | 설명 |
|---|---|---|
| 1 | **시스템 프롬프트** | 기본 설정서 (Anthropic 제공) |

- 작성된 지시는 대화마다 자동 적용 → 반복 설명 불필요

[슬라이드 21] [내용]
# Context ≠ Enforcement

- CLAUDE.md는 시스템 프롬프트가 아니라 **사용자 메시지로 전달**된다
- Claude가 읽고 따르려고 하지만, **모호한 지시는 무시될 수 있다**
- 모호하게 쓰면 무시하고, **구체적으로 쓰면 잘 따른다**

> 맥락(참고 사항)이지 강제(무조건 따름)가 아닙니다. 구체적으로 쓰는 것이 핵심입니다.

# 나쁜 CLAUDE.md vs 좋은 CLAUDE.md

**Bad Pattern 1 — 너무 길다**
- 300줄 이상이면 중요한 규칙이 노이즈에 묻혀서 Claude가 무시하기 시작
- 공식 권장: **200줄 이내**, 실무 권장: **60줄 이내**

**Bad Pattern 2 — 모호한 지시**

| BAD | GOOD |
|---|---|
| "깔끔한 코드를 작성하세요" | "함수는 30줄 이하로 작성하세요" |
| "테스트를 잘 작성하세요" | "새 파일 만들기 전에 나한테 확인하세요" |
| "파일을 잘 정리하세요" | "API 관련 파일은 src/api/ 폴더에 만드세요" |

> 검증 가능한 **구체적인 행동**을 지시해야 합니다

**Bad Pattern 3 — 인덱싱 없이 나열만**
- 규칙을 그냥 나열하면 Claude가 중요도를 판단하지 못함
- 내용이 길어지면 별도 `.md` 파일로 분리하고, CLAUDE.md에서 **참조만** 남기는 것이 효과적
- CLAUDE.md는 **인덱스(목차)** 역할, 상세 내용은 각 파일에

# 컨텍스트란?

**클로드가 "지금 기억하고 있는 것"의 범위**

- 대화 내용, 읽은 파일, 실행한 명령어 — 전부 컨텍스트에 쌓임
- 컨텍스트가 클수록 클로드가 더 정확하게 판단
- 하지만 **무한하지 않음** — 한도(컨텍스트 윈도우)가 있음

**컨텍스트 윈도우 = 한번에 볼 수 있는 크기**
- 현재 최대 1M 토큰 (책 약 5권 분량)
- 대화가 길어지면 앞부분이 자동으로 요약/압축됨
- 너무 많은 파일을 한번에 읽히면 핵심이 묻힐 수 있음

**실전 팁:**
- 한 가지 주제에 집중해서 대화
- 주제가 바뀌면 `/clear`로 새로 시작
- 긴 작업은 단계별로 나눠서 요청

---

[슬라이드 25] [코드]
# /clear — 대화 초기화

주제가 바뀌거나 새로 시작하고 싶을 때:

```
/clear
```

- 현재 대화 내용을 **전부 지우고** 새로운 대화 시작
- CLAUDE.md와 Auto Memory는 유지됨 (프로젝트 설정은 사라지지 않음)
- 이전 대화의 실수나 혼선이 계속 영향을 줄 때 유용

> 비유: 새 동료로 교체 — 단, 업무 매뉴얼(CLAUDE.md)은 그대로

---

[슬라이드 26] [코드]
# /compact — 대화를 압축하기

대화가 길어졌지만 맥락을 유지하고 싶을 때:

```
/compact
```

- `/clear`와 다름 — 대화를 **지우지 않고 요약·압축**
- 컨텍스트 윈도우 공간을 확보하면서 핵심 맥락은 유지
- 긴 작업 중간에 "컨텍스트가 부족합니다" 경고가 뜨면 사용

| 명령어 | 효과 | 비유 |
|--------|------|------|
| `/clear` | 전부 지우고 새로 시작 | 새 동료로 교체 (인수인계 없음) |
| `/compact` | 핵심만 남기고 압축 | 같은 동료에게 핵심만 메모로 전달 |

> 작업 중이라면 `/clear` 대신 `/compact`가 더 안전합니다

[슬라이드 27] [내용]
# Auto Memory — 대화를 넘어서 기억하기

**대화가 끝나도 중요한 정보를 기억하는 기능**

- 클로드가 대화 중 중요한 정보를 자동으로 저장
- 다음 대화에서도 이전에 배운 내용을 활용
- 저장 위치: `~/.claude/` 폴더 내 메모리 파일

**기억하는 것들:**
- 사용자 선호 (언어, 스타일, 작업 방식)
- 프로젝트 맥락 (진행 상황, 결정 사항)
- 피드백 ("이렇게 하지 마" → 다음부터 반영)

**직접 요청도 가능:**
```
"앞으로 모든 응답은 한국어로 해줘 — 이걸 기억해줘"
```

> CLAUDE.md는 **프로젝트의 매뉴얼**, Auto Memory는 **클로드의 업무 일지**

# 컨텍스트 관리 전략 정리

| 도구 | 역할 | 지속 범위 |
|------|------|----------|
| **컨텍스트** | 현재 대화의 작업 기억 | 이번 대화만 |
| **/clear** | 대화 초기화 | - |
| **/compact** | 대화 압축 (맥락 유지) | 이번 대화만 |
| **CLAUDE.md** | 프로젝트 규칙·매뉴얼 | 항상 (파일) |
| **Auto Memory** | 학습된 선호·피드백 | 항상 (자동 저장) |

**실전 흐름:**
1. 프로젝트 시작 → CLAUDE.md로 기본 규칙 설정
2. 작업 중 → 컨텍스트가 쌓임
3. 컨텍스트 부족 → `/compact`로 압축
4. 주제 전환 → `/clear`로 초기화
5. 중요한 패턴 → Auto Memory가 자동 저장

> 이 5가지를 이해하면 클로드와 **장기적으로 효율적인 협업**이 가능합니다

# 프롬프트 엔지니어링 vs 컨텍스트 엔지니어링

| | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|--|--|--|
| **한마디로** | AI에게 **좋은 질문**을 하는 기술 | AI가 **잘 일할 환경**을 만드는 기술 |
| **초점** | "어떻게 물어볼까?" | "무엇을 미리 알려줄까?" |
| **예시** | 프롬프트 템플릿, 역할 부여, 출력 형식 지정 | CLAUDE.md, Memory, 파일 참조, /compact |
| **효과 범위** | 그 질문 한 번 | 한번 세팅하면 계속 효과 |
| **우리와의 관계** | 나비프로젝트에서 해온 전문 영역 | 지금 이 강의에서 배우는 내용 |

**프롬프트 엔지니어링** — "어떻게 하면 결과가 잘 나올까?"
**컨텍스트 엔지니어링** — "어떤 정보를 줘야 결과가 잘 나올까?"

> 프롬프트 엔지니어링이 **말을 잘 거는 기술**이라면,
> 컨텍스트 엔지니어링은 **일잘하는 환경을 세팅하는 기술**입니다

# Shift+Tab — 클로드의 자유도 조절

**입력창에서 `Shift+Tab`을 누르면 3가지 모드가 순환됩니다:**

| 모드 | 표시 | 동작 |
|------|------|------|
| **기본** | - | 파일 수정/명령 실행마다 "할까요?" 물어봄 |
| **자동 편집** | ⏵⏵ accept edits | 파일 편집은 자동 승인, 명령 실행은 물어봄 |
| **플랜 모드** | ⏸ plan mode | 읽기만 가능 — 편집/실행 안 함 (계획만 세움) |

**실전 활용:**
- 처음엔 **기본 모드**로 하나하나 확인하며 작업
- 클로드가 믿을만하면 **자동 편집**으로 전환해서 속도 UP
- 큰 작업 전에 **플랜 모드**로 "어떻게 할 건지 먼저 알려줘"

> 비유: 신입 직원에게 주는 권한 — "물어봐" → "글은 써도 돼" → "계획만 세워"

# @ 파일 참조 & 이미지 붙여넣기

**@ 파일 참조 — "이 파일 좀 봐"**
```
@README.md 이 파일 내용 요약해줘
@src/data/ 이 폴더 구조 분석해줘
```
- 대화 중에 특정 파일/폴더를 빠르게 지정
- 파일명 일부만 입력해도 자동완성

**이미지 붙여넣기 — 스크린샷으로 대화**
- `Ctrl+V` — 클립보드 이미지 바로 붙여넣기
- 에러 화면 캡처 → "이 에러 뭐야?" 가능
- UI 디자인 이미지 → "이거랑 비슷하게 만들어줘"

> 텍스트로 설명하기 어려우면, 그냥 **보여주세요**

# Model 전환 — 상황에 맞는 두뇌 선택

| 모델 | 특징 | 언제 쓰나 |
|------|------|----------|
| **Opus** | 가장 똑똑, 가장 비쌈 | 복잡한 분석, 중요한 작업 |
| **Sonnet** | 균형잡힘 | 일반적인 작업 대부분 |
| **Haiku** | 가장 빠름, 가장 저렴 | 간단한 질문, 빠른 확인 |

**실전 전략:**
- 기본은 Sonnet으로 작업
- 복잡한 판단이 필요하면 Opus로 전환
- 단순 확인은 Haiku로 토큰 절약

> "모든 작업에 가장 비싼 모델"은 낭비입니다

# Effort — 얼마나 깊이 생각할까?

| 레벨 | 설명 | 토큰 소모 |
|------|------|----------|
| **Low** (●○○) | 간단한 작업, 빠른 답변 | 적음 |
| **Medium** (●●○) | 표준 추론 (기본값) | 중간 |
| **High** (●●●) | 깊이 생각하고 답변 | 높음 |
| **Max** (●●●●) | 최대 추론, 제약 없이 사고 | 매우 많음 |

**실전 전략:**
- 간단한 질문 → Low
- 일반 작업 → Medium (기본값)
- 복잡한 분석, 설계 → High 또는 Max

> Effort = "이 작업에 얼마나 시간을 들여라"라는 지시

# Thinking — 사고 과정 보기

**ON으로 켜면:** 클로드가 답변 전에 "생각하는 과정"을 보여줌

```
🤔 Thinking...
- 사용자가 요청한 것은 엑셀 데이터 정리
- 파일이 3개 있으니 먼저 구조 확인
- 날짜 형식이 다르니 통일 필요
```

**언제 켜나?**
- 클로드가 왜 그런 결정을 했는지 알고 싶을 때
- 복잡한 작업에서 방향이 맞는지 확인할 때
- 학습 목적 — 클로드의 사고방식 이해

**언제 끄나?**
- 단순 반복 작업
- 빠른 응답이 필요할 때

> Thinking ON = 토큰을 더 쓰지만, 더 정확한 결과

# 되돌리기

클로드 코드는 **실제로 파일을 수정**하지만, 되돌리기가 가능합니다

**기본 안전장치: Rewind**
- 클로드가 수정한 내용은 **Rewind**로 되돌릴 수 있음
- 하지만 **파일 삭제** 등은 Rewind로도 복구가 어려움

**이럴 땐 Git이 필요합니다**
- Git으로 커밋해두면 어떤 상황이든 복구 가능

# Git이란?

**한 마디로: 파일의 "저장 이력"을 관리하는 도구**

**작업 폴더의 스냅샷을 찍어두는 도구**

| 스냅샷 | Git |
|--------|-----|
| 폴더 상태를 통째로 찍어둠 | **커밋** — 지금 상태를 저장 |
| 스냅샷이 계속 쌓임 | 언제든 원하는 시점으로 이동 |
| "3번째 사진으로 돌아가줘" | **"되돌려줘"** 한 마디면 복구 |

- 예전: 명령어 공부 + 별도 툴 설치 필요
- 지금: **"커밋해"**, **"되돌려줘"** 자연어 한 줄이면 끝
