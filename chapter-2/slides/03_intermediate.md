[슬라이드 43] [내용]
# Skill이란?

**AI에게 주는 "업무 매뉴얼" — 한 번 만들면 자동으로 적용**

신입 사원에게 업무 매뉴얼을 주면 일정한 품질의 결과물이 나오듯,
Skill을 만들어두면 클로드가 **매번 같은 품질**로 작업합니다.

**프롬프트와의 차이:**

| | 프롬프트 | Skill |
|---|---------|-------|
| **사용** | 매번 직접 입력 | 한 번 만들면 자동 트리거 |
| **재사용** | 복붙 필요 | 이름만 부르면 실행 |
| **공유** | 어려움 | 파일 하나로 팀 공유 |

**핵심 장점:**
- 반복 작업 자동화
- 결과물 품질 일정하게 유지
- 팀원과 쉽게 공유

> Skill = "내가 자주 시키는 일을 미리 적어둔 지시서"

---

[슬라이드 44] [내용]
# Skill의 구조 — 두 단계 로딩

**skill.md 파일이 진입점, 참조 파일과 함께 디렉토리 단위로 관리**

```
.claude/skills/
  └─ proposal/
       ├─ skill.md        ← 진입점 (설정 + 프롬프트)
       └─ template.md     ← 참조 파일 (양식 등)
```

**똑똑한 로딩 방식 (2단계):**

| 단계 | 로딩 내용 | 컨텍스트 부담 |
|------|----------|-------------|
| **1단계** | Description(설명)만 로딩 | 매우 가벼움 |
| **2단계** | 필요할 때 Full Prompt 로딩 | 필요한 만큼만 |

- CLAUDE.md는 **항상** 로딩됨 (무조건 컨텍스트 차지)
- Skill은 **필요할 때만** 로딩 → 컨텍스트 부담이 적음

> 비유: CLAUDE.md = 항상 펼쳐놓은 매뉴얼, Skill = 필요할 때 꺼내 보는 매뉴얼

- AI가 "이 Skill을 쓸지 말지" 판단하는 필드
- 사용자가 실제로 말할 법한 표현을 포함해야 함

[슬라이드 49] [내용]
# MCP란?

**Model Context Protocol — 클로드와 외부 도구를 연결하는 표준**

클로드 코드 혼자서는:
- 노션 페이지 수정 ❌
- 구글 시트 수정 ❌
- 슬랙 메시지 전송 ❌

MCP를 연결하면:
- 노션 페이지 수정 ✅
- 구글 시트 수정 ✅
- 슬랙 메시지 전송 ✅

> 비유: 스마트폰에 앱을 설치하는 것. MCP = 클로드의 "앱 스토어"

---

[슬라이드 50] [코드]
# MCP 연결하기

**클로드에게 그냥 말하면 됩니다:**

```
"노션 MCP를 연결해줘"
```

클로드가 자동으로:
1. 필요한 MCP 서버 파악
2. `.mcp.json` 설정 파일 생성
3. 연결 테스트

**수동 설정도 가능:**
```json
// .mcp.json
{
  "mcpServers": {
    "notion": {
      "command": "npx",
      "args": ["@notionhq/notion-mcp-server"],
      "env": { "NOTION_API_KEY": "your-key" }
    }
  }
}
```

> 자연어 요청이 안 되면 수동 설정도 알아두면 좋습니다

# ⚠️ MCP 설치 전략 — 적게, 핵심만

**MCP를 많이 설치하면 안 되는 이유:**
- 설치된 MCP의 **모든 도구 설명이 시작 시 로딩** → 컨텍스트 압박
- 도구가 많을수록 클로드가 **잘못된 도구를 선택**할 확률 증가
- 최근 트렌드: MCP 대신 **Skill + Sub-Agent 조합**으로 대체하는 경우가 늘고 있음

**권장 전략:**

| 전략 | 설명 |
|------|------|
| **핵심 1~2개만 설치** | 매일 쓰는 서비스만 MCP로 연결 (예: Notion만) |
| **프로젝트별 분리** | `.mcp.json`을 프로젝트마다 다르게 설정 |
| **대체 가능하면 Skill로** | 단순 API 호출은 Skill로 대체가 더 가벼움 |
| **필요할 때만 활성화** | 안 쓰는 MCP는 설정에서 비활성화 |

> "앱을 100개 설치한 스마트폰"보다 **"필요한 앱 3개만 설치한 스마트폰"**이 빠릅니다

[슬라이드 55] [내용]
# Hook이란?

**이벤트가 발생하면 자동으로 액션을 실행하는 자동화 엔진**

```
이벤트 발생 → 훅 감지 → 조건 검사 → 액션 실행
```

예시:
- 파일 저장할 때 → 자동 포맷팅
- 도구 호출 전 → API 키 포함 여부 체크
- 클로드 응답 대기 시 → 알림 전송
- 작업 종료 시 → 결과 보고서 생성

**비유:**
- 스마트홈의 "현관문 열리면 → 조명 켜기"
- 이메일의 "특정 발신자 → 자동 폴더 분류"

> Hook = "조건 → 동작"의 자동화 규칙

---

[슬라이드 56] [내용]
# Hook의 4가지 이벤트 타입

| 이벤트 타입 | 언제 실행? | 용도 |
|------------|-----------|------|
| **PreToolUse** | 도구 호출 **직전** | 입력값 검증, 승인, 차단, 수정 |
| **PostToolUse** | 도구 실행 **직후** | 결과 검증, 린트, 포맷팅 |
| **Notification** | 사용자 응답 대기 시 | 알림 전송, 로깅, 외부 연동 |
| **Stop** | 에이전트 턴 종료 시 | 최종 정리, 보고서 생성, 상태 저장 |

**작동 흐름 예시 (파일 쓰기):**
```
클로드가 파일 쓰기 시도
  → PreToolUse: "API 키 포함?" 체크 → 통과
  → 파일 쓰기 실행
  → PostToolUse: 자동 포맷팅 실행
```

> 훅 실행 중에는 클로드가 대기합니다 — 오래 걸리는 작업은 타임아웃 설정을 권장

---

[슬라이드 57] [코드]
# Hook 설정하기

**방법 1: `/hooks` 명령어로 확인·관리**
```
/hooks     ← 현재 설정된 훅 목록 확인
```

**방법 2: 클로드에게 요청**
```
"파일 저장할 때마다 자동으로 포맷팅하는 Hook 만들어줘"
"클로드가 응답 대기 중이면 알림을 보내는 Hook 추가해줘"
```

**방법 3: settings.json 직접 수정**
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "command": "echo '파일 쓰기 감지됨'"
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "command": "echo '클로드가 응답을 기다리고 있습니다'"
      }
    ]
  }
}
```

> 생성된 훅은 settings.json에 저장되며, 이벤트 타입·매칭 조건·실행 액션으로 구성됩니다

# Hook 활용 예시

**1. API 키 유출 차단 (PreToolUse)**
- 클로드가 파일을 쓰기 직전 → API 키/비밀번호 포함 여부 검사
- 예: 나라장터 API 키가 코드에 하드코딩되면 쓰기 자체를 차단

**2. 구글시트 기록 후 검증 (PostToolUse)**
- MCP로 구글시트에 공고를 추가한 직후 → 행 번호·데이터 정합성 로그 기록
- 아까처럼 엉뚱한 행에 들어가는 실수를 자동 감지

**3. 검색 완료 알림 (Notification)**
- `/bid-search`로 123건 크롤링 중 → 클로드가 대기할 때 알림음 전송
- 다른 일을 하다가도 "검색 끝났어요" 알림으로 바로 복귀

**4. 시트 기록 후 메일 제안 (PostToolUse)**
- Google Sheets MCP로 기록 완료 → "메일로 보내드릴까요?" 자동 제안
- 승인하면 이메일 MCP로 요약 메일 발송까지 연결

> Skill이 "무엇을" 하고, MCP가 "어디에" 연결하고, Hook이 "언제" 자동 실행할지 결정합니다
